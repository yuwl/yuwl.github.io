14.2 并发同步工具类

Yuwl 2024-08-11 13:28:48
Categories： Java Tags： 多线程

现在我们来学习 Java 中并发包 (java.util.concurrent)中的同步工具类。

ReentrantLock

描述: ReentrantLock是Lock接口的一个常用实现，它允许显式地锁定和解锁。

tryLock() 方法

tryLock() 方法

描述: tryLock()方法尝试获取锁，但不会无限期地等待。如果锁不可用，它会立即返回false。

ReentrantLock 和 synchronized的区别

  1. 灵活性
    ReentrantLock 比 synchronized 更加灵活。ReentrantLock 提供了 tryLock() 方法,允许线程尝试获取锁而不会被阻塞。如果锁不可用,tryLock() 会立即返回 false。这使得开发者可以更好地控制获取锁的策略,减少死锁的风险。
  2. 公平性
    ReentrantLock 支持公平锁,即优先分配给等待时间最长的线程。而 synchronized 不保证任何特定的获取顺序。
  3. 中断和超时
    使用 ReentrantLock 的 lockInterruptibly() 方法,线程在等待锁时可以被中断。synchronized 不支持中断。ReentrantLock 还支持带超时的获取锁,而 synchronized 会无限期等待。
  4. 条件变量
    ReentrantLock 提供了 Condition 接口,允许更精细的线程协调。synchronized 使用 Object 的 wait(), notify() 和 notifyAll() 方法实现线程等待和唤醒。
  5. 性能
    在高竞争的情况下,ReentrantLock 可能会比 synchronized 有更好的性能,因为它避免了一些不必要的阻塞。但在低竞争的情况下,synchronized 可能会更快。
  6. 易用性
    synchronized 使用更简单,不需要显式地获取和释放锁。而 ReentrantLock 需要手动调用 lock() 和 unlock() 方法,更容易出错

Condition

使用Lock和Condition实现生产者-消费者模式。

使用 Lock 和 Condition 可以更灵活地控制线程的等待和唤醒。与传统的 synchronized 和 wait()/notify() 机制相比，Lock 和 Condition 提供了更加细粒度的控制，同时也允许多个条件变量从而实现更加复杂的同步逻辑。

ReadWriteLock

ReadWriteLock 是一种用于控制并发访问共享资源的锁机制。它允许同时允许多个线程读取共享资源，但只允许一个线程进行写操作。

ReadWriteLock 接口提供了两个锁对象：读锁和写锁。读锁用于并发读取共享资源，而写锁用于独占访问共享资源。

1. 基本概念

  • 读锁 (ReadLock)：允许多个线程同时读取数据，而不需要相互等待。只要没有线程持有写锁，读锁就可以被多个线程同时获取。
  • 写锁 (WriteLock)：写操作需要独占锁，也就是说，当一个线程持有写锁时，其他线程（无论是读还是写）都必须等待。

2. ReadWriteLock 的实现类

Java 中的 ReadWriteLock 有一个常用的实现类：ReentrantReadWriteLock。

ReentrantReadWriteLock：一种可重入的读写锁，支持公平和非公平的锁获取顺序。

3. 使用 ReadWriteLock 的步骤

  • 创建 ReentrantReadWriteLock 实例：通过 ReentrantReadWriteLock 类创建读写锁的实例。
  • 获取读锁或写锁：根据需要，通过 readLock() 或 writeLock() 方法获取锁。
  • 执行操作：在获取锁后，执行读或写操作。
  • 释放锁：操作完成后，必须手动释放锁。

使用场景

ReadWriteLock 非常适合以下场景：

  • 读操作远多于写操作：例如缓存系统、配置读取等场景。
  • 需要频繁读取而偶尔写入的资源：例如日志记录、状态监控等。

StampedLock

StampedLock 是 Java 8 引入的一种用于高并发场景下的锁机制。它提供了读写锁的功能，同时引入了乐观读锁的概念。与 ReentrantReadWriteLock 相比，StampedLock 在读多写少的场景下性能更好。

1. 基本概念

  • 写锁 (write lock)：与传统的写锁类似，当线程持有写锁时，其他线程不能获取读锁或写锁。
  • 读锁 (read lock)：与传统的读锁类似，多个线程可以同时持有读锁，但当有写锁时，读锁将被阻塞。
    乐观读锁 (optimistic read lock)：这是 StampedLock 的一个新特性，它允许线程在不真正加锁的情况下读取数据。乐观读锁不会阻塞写操作，但在使用时需要验证数据的一致性。

2. StampedLock 的操作

  • 获取和释放锁：StampedLock 通过 stamp（邮票）来标记锁的状态，获取锁时会返回一个长整型的 stamp，释放锁时需要提供这个 stamp。
  • 验证乐观读锁：乐观读锁在获取后，需要通过 validate() 方法验证在此期间是否有写操作发生。

3. 使用 StampedLock 的步骤

  • 创建 StampedLock 实例：通过 new StampedLock() 创建锁实例。
  • 获取锁：使用 writeLock(), readLock() 或 tryOptimisticRead() 获取锁，并保存返回的 stamp。
  • 执行操作：在获取锁后，执行读或写操作。
  • 释放锁：通过 unlockWrite(), unlockRead() 或 validate() 方法释放锁或验证锁的有效性。

Semaphore

Semaphore 类：信号量，用于控制并发访问的线程数量。Semaphore 通常用于限制某些资源的并发访问，比如连接池、数据库连接等。

1. 基本概念

  • 信号量 (Semaphore)：Semaphore 维护一个计数器，用来表示可用的资源数量。线程可以通过 acquire() 方法获取资源（减少计数器），通过 release() 方法释放资源（增加计数器）。
  • 许可 (Permit)：Semaphore 的计数器实际上代表了可用的“许可”数量。获取一个许可相当于减少计数器的值，释放一个许可则相当于增加计数器的值。

2. Semaphore 的构造方法

  • Semaphore(int permits)：创建一个具有指定许可数的 Semaphore。
  • Semaphore(int permits, boolean fair)：创建一个具有指定许可数的 Semaphore，并指定是否使用公平策略。公平策略确保线程按请求许可的顺序获取许可。

3. 使用 Semaphore 的步骤

  • 创建 Semaphore 实例：通过 new Semaphore(permits) 创建信号量实例。
  • 获取许可：通过 acquire() 方法获取许可。如果许可不足，线程会被阻塞直到有许可可用。
  • 执行操作：获取许可后，执行受控的资源操作。
  • 释放许可：通过 release() 方法释放许可，使得其他线程可以获取许可。

CountDownLatch

CountDownLatch 类：倒计数器，用于让一个或多个线程等待其他线程完成。

1. 基本概念

  • 计数器 (CountDownLatch)：CountDownLatch 维护一个初始值为给定数量的计数器。每次调用 countDown() 方法时，计数器减 1。当计数器减为零时，所有等待在 await() 方法上的线程都会被唤醒。
  • 等待线程：线程调用 await() 方法进行等待，直到计数器为零。

2. CountDownLatch 的构造方法

  • CountDownLatch(int count)：创建一个 CountDownLatch 实例，并设置初始计数器值。

3. 使用 CountDownLatch 的步骤

  • 创建 CountDownLatch 实例：通过 new CountDownLatch(count) 创建一个具有指定计数器的 CountDownLatch。
  • 执行并发任务：启动多个线程执行任务，并在任务完成后调用 countDown() 方法减少计数器。
  • 等待计数器归零：主线程或其他线程调用 await() 方法等待，直到计数器变为零。

CyclicBarrier

CyclicBarrier 类：类似于 CountDownLatch，但可以重用，允许一组线程互相等待，直到达到共同的屏障点。

1. 基本概念

  • 屏障 (Barrier)：CyclicBarrier 设置了一个屏障点，当指定数量的线程（称为“参与者”）都到达这个屏障时，所有线程会同时继续执行。
  • 重用性：与 CountDownLatch 不同，CyclicBarrier 在所有线程释放后，可以被重置并再次使用。

2. CyclicBarrier 的构造方法

  • CyclicBarrier(int parties)：创建一个 CyclicBarrier，指定参与的线程数。
  • CyclicBarrier(int parties, Runnable barrierAction)：创建一个 CyclicBarrier，指定参与的线程数，并提供一个在所有线程都到达屏障点时要执行的 Runnable 动作。

3. 使用 CyclicBarrier 的步骤

  • 创建 CyclicBarrier 实例：通过 new CyclicBarrier(parties) 创建一个 CyclicBarrier 实例，指定参与的线程数。
  • 线程等待：每个线程在执行任务的过程中调用 await() 方法，等待其他线程到达屏障点。
  • 继续执行：当所有线程都调用 await()，到达屏障点后，所有线程同时继续执行。
